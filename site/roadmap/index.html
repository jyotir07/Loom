    Loom — Project Description & Development Phases¶

    Project description¶

    Loom is a Python framework that sits between an organization's projects and the AI providers they call. Instead of each project re-integrating OpenAI, Anthropic, Gemini, and a dozen others — each with its own SDK, retry logic, key handling, and cost surface — every project makes one kind of call into Loom, and Loom handles vendor selection, model routing, caching, batching, retries, key management, and cost logging.

    Loom is not a lowest-common-denominator aggregator. Each vendor is integrated with its own native SDK so vendor-specific capabilities (prompt caching, grounding, image polling, streaming, structured output) are preserved instead of flattened away.

    The framework is built on top of the existing Models Catalog project, which already provides the provider abstraction, native SDK adapters, a unified model catalog, and a working generate(modality, model, params, prompt) -> dict contract. The remaining work is to package that engine as an installable library, layer cost-optimization features on top, and ship observability and documentation around it.

    Goals¶

    • One contract — every project calls loom.generate(...); nothing else.
    • One place for upgrades — new providers, new models, and new optimizations land in Loom and every consumer inherits them.
    • One audited key store — vendor API keys live in the Loom deployment, not in N project repos.
    • One cost view — every call is logged with provider, model, latency, tokens, and cost; finance gets a unified dashboard.
    • No framework lock-in — Loom is a library, callable from Flask/FastAPI/Django/Celery/Lambda/CLI/notebooks.

    Non-goals¶

    • Replacing vendor SDKs with a homogenized API surface.
    • Becoming a managed service (though Loom can be deployed as one if a team wants an internal HTTP layer).
    • Acting as a model router that hides which vendor served a call — Loom always reports the provider and model used.

    Phases of development¶

    Each phase delivers independently usable value. An org can stop at Phase 2 and have a useful internal library; Phase 3 is where the cost-optimization story compounds.

    Phase 0 — Existing foundation (already in place)¶

    The seed project this work builds on. Captured here so the starting line is explicit.

    What exists today:

    • Flask app (app.py) that exposes the catalog and generate(...) over a web UI.
    • models_catalog.py with stable model IDs, upstream model IDs, default params, and pricing.
    • Native-SDK provider adapters for 14+ vendors (OpenAI, Anthropic, Gemini, xAI, Mistral, DeepSeek, MiniMax, Z.AI, Perplexity, Together, BFL, Seedream, Hunyuan, Ideogram).
    • Shared adapter pattern for OpenAI-compatible vendors (~10 lines to add a new one).
    • Unified response shape: {"kind": "text" | "image", ...}.
    • Templates, static assets, and a seed script (seed_db.py) for local exploration.

    Status: working, but coupled to the Flask app — not yet importable as a standalone library.

    Phase 1 — Package extraction (1–2 weeks)¶

    Goal: turn the engine inside app.py into an installable Python package without breaking the existing Flask app.

    Deliverables:

    • loom/ package with a clean public API: loom.generate(...), loom.Loom, loom.Catalog.
    • Provider registry and adapters moved out of the Flask app into loom/providers/.
    • Catalog moved into loom/catalog/ with an in-memory default backend.
    • pyproject.toml / setup.cfg and a publishable wheel (pip install loom).
    • Flask app refactored to import from the new package — UI keeps working, now as a thin consumer of the library.
    • Loom.from_env() constructor that reads vendor API keys from environment variables.
    • Smoke tests covering one text provider and one image provider end-to-end.

    Exit criteria: a fresh Python project can pip install loom, call loom.generate(...), and get a response back. The existing Flask app still runs unchanged from the user's perspective.

    Phase 2 — Library hardening (3–4 weeks)¶

    Goal: make Loom usable as a real internal dependency for production projects.

    Deliverables:

    • Programmatic configuration — Loom(catalog=..., api_keys=..., cache_backend=...) for projects that don't want env-var-based wiring.
    • Pluggable catalog backends — in-memory (default), YAML file, and Postgres. Same interface, swap at construction time.
    • Typed responses — TypedDict / Pydantic models for TextResponse and ImageResponse so IDE autocomplete works and downstream code can result["text"] safely.
    • Async support — agenerate(...) mirroring the sync API, plus AsyncLoom class. Works inside FastAPI, Celery, and serverless runtimes.
    • Per-call cost field — every response includes cost_usd and the configured local-currency equivalent.
    • Structured logging — every call logs provider, model, latency, input/output tokens, and cost in a parseable format.
    • Error taxonomy — LoomError, ProviderError, RateLimitError, AuthError, ModelNotFoundError — so consumers can except cleanly.
    • Documentation — README, API reference, and a "migrating from direct vendor SDKs" guide.
    • Internal release — published to the org's private package index.

    Exit criteria: an internal project can migrate one call site from a direct vendor SDK to Loom in under an hour, with no loss of functionality and gained cost logging.

    Phase 3 — Optimization & observability layer (ongoing)¶

    Goal: make Loom pay for itself. Every optimization built here lands once and every consumer inherits it on upgrade.

    Deliverables (each independently shippable):

    • Response cache — identical (provider, model, prompt, params) calls hit Redis/Memcached instead of the API. Per-call and global toggles. Target: 20–60% savings on workloads with repeated queries.
    • Vendor-native prompt caching — wire up Anthropic, OpenAI, Gemini, and DeepSeek prompt-cache headers for repeated system prompts and few-shot examples. Target: 50–90% discount on cached prefix tokens.
    • Smart model routing — try a cheap model first (Haiku, GPT-4o-mini, Gemini Flash); escalate only when confidence is low or validation fails. Target: 50–80% savings on mixed workloads.
    • Batch API integration — auto-batch non-urgent calls to OpenAI/Anthropic/Gemini batch endpoints (50% discount, 24h turnaround). Async-friendly handles.
    • Retry & failover — exponential backoff done correctly once; if one vendor is down, fall back to an equivalent model on another vendor instead of failing.
    • Request deduplication — collapse identical concurrent calls into one upstream request, fan the response back out.
    • Observability dashboard — per-project, per-model, per-day cost reporting; budget alerts; visibility into which optimizations are actually saving money.
    • Centralized key vault integration — pluggable backends for AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault.
    • Semver stability commitment — public API frozen, breaking changes only on major versions.
    • Public docs site — full reference, tutorials, migration guides, optimization cookbook.

    Exit criteria: this phase is open-ended. Each optimization is gated behind a flag and rolled out independently, with before/after cost measurements published to the dashboard.

    What ships when¶

    Phase Duration Cumulative outcome
    0 (done) Working Flask app + provider adapters + catalog
    1 1–2 wks Installable loom package; Flask app uses it
    2 3–4 wks Production-ready internal library with async, typed responses, cost logging
    3 ongoing Cost-optimization layer, observability, public docs

    Phases are sequential — Phase 2 depends on Phase 1's package boundary, and Phase 3's optimizations hook into Phase 2's logging and async paths.
